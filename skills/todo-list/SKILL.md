# todo-list

Regenerates `docs/todo-list.md`: a durable, re-runnable inventory of actionable comment markers
left in the code. IDs are stable across runs — once a task is assigned an ID, re-running this
skill later must not reassign that number to something else, so "let's fix #7" keeps meaning the
same thing next week.

## 1. Read the previous inventory, if any

Read `docs/todo-list.md` if it exists. Parse out:

- Its `<!-- todo-list: next-id=N -->` counter comment (on the first line) — the next ID to hand
  out. If the file doesn't exist yet, start at `next-id=1`.
- Every existing entry: ID, Type, Description, and its references.

This is what makes IDs stable — never invent a fresh ID for a task that's just a re-scan of one
already in this file.

## 2. Scan the codebase for markers

`Grep` the whole repo (excluding `bin/`, `build/`, `vendor/`, `node_modules/`, and any other
generated/vendored output) for comment lines containing, case-insensitively, one of these marker
keywords, with or without a leading `@`: `TODO`, `FIXME`, `BUG`, `FIX`, `HACK`, `XXX`,
`OPTIMIZE`, `REFACTOR`, `DEPRECATED`. Don't include plain `NOTE`/`INFO`-style comments — those
aren't actionable markers. For each hit, capture the file, line number, marker keyword
(normalize to canonical uppercase, e.g. `todo:` and `@TODO` both become `TODO`), and the comment
text itself (strip the comment syntax — `//`, `#`, `/* */`, `--`, etc. — and the marker/any
`(username)`/`:` around it).

If the repo is large, delegate the sweep to the `Explore` agent instead of grepping ad hoc
file-by-file.

## 3. Cluster hits into tasks

Multiple hits are **the same task** (one table entry, multiple references) when their comment
text is the same or near-identical (allowing minor wording/whitespace differences) — this
commonly happens when the same TODO was copy-pasted across similar files (e.g. repeated test
fixtures). Otherwise, each distinct comment is its own task, even if two tasks share a marker
type or live in the same file.

## 4. Match clustered tasks against the previous inventory

For each clustered task from step 3, check step 1's existing entries:

- **Matches an existing entry** (shares at least one identical `file:line` reference, or has
  near-identical description text) — keep that entry's ID; update its description/references if
  they changed (e.g. the comment moved lines, or spread to/dropped from a file).
- **No match** — it's new; assign it the counter's current `next-id` value, then increment the
  counter.
- An existing entry with **no matching task found this scan** — its comment was resolved/removed
  from the code. Drop it from the table (don't keep stale entries), but note it in the step 6
  report. Never reuse its ID for anything else — the counter only ever increases.

## 5. Group into themes

Assign each task a theme — a short, human-meaningful grouping (e.g. "Error handling", "Test
coverage", "Performance", "API surface gaps"), inferred from what the comment is actually about,
not from its marker type (a `TODO` and a `FIXME` can belong to the same theme; two `TODO`s can
belong to different ones). Re-running this skill may reclassify a task's theme if the grouping
judgment changes — that's fine, since identity (step 4) is tracked by content/references, not by
theme label.

## 6. Write `docs/todo-list.md`

- First line: `<!-- todo-list: next-id=N -->` with the counter's new value.
- A one-line intro noting this file is generated by `/todo-list` and when it was last run.
- One `### <Theme>` heading per theme, each followed by a table:
  `ID | Type | Description | References` — `References` lists every `file:line` for that task
  (comma-separated, or one per line if there are many), Type is the normalized marker keyword.
- Order themes and rows however reads best (e.g. themes by task count descending) — there's no
  required ordering beyond "grouped by theme."

Overwrite whatever is currently at `docs/todo-list.md`. Create `docs/` if it doesn't exist yet.

## 7. Report

Summarize in chat: total tasks found, new IDs assigned (with their numbers), and any prior
entries dropped because their comment no longer exists in the code (with their old numbers, so
the user can tell what disappeared and double check it was actually resolved, not just moved
somewhere the scan missed).
